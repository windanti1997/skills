# 代码层过拟合专项检查清单

用于审查代码实现、技术方案、测试代码时的专项过拟合检查。

## 一、测试真实性检查

### 1.1 断言深度
- [ ] 测试断言验证的是**具体输出内容**，而不是只检查"函数存在/文件生成/返回非空
- [ ] 断言检查了输出的**关键属性**，而不只是类型或长度
- [ ] 有**负向断言**（不该发生的事确实没发生）
- [ ] 断言不是 `expect(true).toBe(true)` 这种永远通过的假测试

### 1.2 测试与实现的关系
- [ ] 测试是**先有期望行为再写测试**，而不是反过来根据实现写测试
- [ ] 实现改动时，是**修复实现**让测试通过，而不是修改测试适配实现
- [ ] 测试用例是从**需求/场景**推导出来的，不是从实现代码反推的
- [ ] 如果删掉核心逻辑，测试会失败（可以做反向验证）

### 1.3 测试覆盖质量
- [ ] 覆盖了**边界条件**（空值、最大值、最小值、零值）
- [ ] 覆盖了**错误路径**（异常输入、依赖失败、权限不足）
- [ ] 覆盖了**并发/时序**问题（如果是并发系统）
- [ ] 测试数据是**真实场景**的，不是为了通过测试而构造的简单样例

### 1.4 测试代码质量
- [ ] 测试有**明确的测试目的**，每个测试只测一件事
- [ ] 测试描述清晰说明了**测什么、期望什么**
- [ ] 没有大量重复代码，有合理的 test fixture / helper
- [ ] 测试是可重复运行的，不依赖外部状态

## 二、测试完整性检查

### 2.1 测试分层
- [ ] 有**单元测试**：验证单个函数/类的逻辑
- [ ] 有**集成测试**：验证模块之间的交互
- [ ] 有**端到端测试**（如果是关键路径）：验证完整用户流程
- [ ] 各层测试比例合理，不是只有单测没有集成

### 2.2 功能集成验证
- [ ] 涉及多个模块的功能，**模块拼起来真的能工作
- [ ] 接口契约测试：API 的输入输出格式符合预期
- [ ] 数据流向正确：从输入到输出的完整链路验证过
- [ ] 不是"每个单元都对，但合起来不对"的情况

### 2.3 非功能需求验证
- [ ] 性能：声称的性能指标有 benchmark 数据支持
- [ ] 可靠性：错误处理、重试、降级机制有验证
- [ ] 安全性：注入、权限、数据校验有测试
- [ ] 可观测性：日志、监控、告警是真实有效的

## 三、代码真实性检查

### 3.1 功能完整性
- [ ] 功能是**真实现**了，不是 stub / placeholder / TODO
- [ ] 核心逻辑有实质内容，不是只有函数壳子
- [ ] 声称支持的场景确实都覆盖了，不是只写了主路径
- [ ] 配置项、参数、选项都是真实可用的，不是摆设

### 3.2 错误处理真实性
- [ ] try-catch 里有**真实的错误处理**，不是吞掉错误
- [ ] 错误有明确的分类和传递路径，不是随便返回 null / 空字符串
- [ ] 失败场景有降级或重试策略，不是直接崩或直接吞
- [ ] 错误信息对调试有帮助，不是"出错了"三个字

### 3.3 代码与文档一致性
- [ ] 代码行为和文档描述一致
- [ ] 配置项说明和实际默认值一致
- [ ] 接口文档和实际接口行为一致
- [ ] README 里的例子能直接跑通

### 3.4 性能与优化真实性
- [ ] 声称的优化有**实测数据**支持，不是拍脑袋
- [ ] 优化前后有对比数据
- [ ] 没有为了"看起来专业"而加不必要的优化
- [ ] 复杂度分析是准确的，不是瞎写 O(1) 实际是 O(n²)

## 四、Reviewer 自我审查（防放水检查）

在给出审查结论前，先问自己：

1. 我有没有跑过测试/看过测试运行结果？还是只看了测试代码？
2. 我有没有验证过功能真的能用吗？还是只看了代码结构？
3. 我是不是因为"代码量很大"就觉得应该没问题？
4. 我找的问题是真问题，还是风格偏好？
5. 最关键的那个风险点我检查了吗？
6. 如果这是生产代码，我愿意为它负责吗？
7. 我的 PASS/FAIL 结论有明确的证据支持吗？
8. 我是不是应该叫个独立 subagent 来做 code review，而不是自己审？

## 五、什么时候必须叫 Subagent 做 Code Review

代码审查是最容易自我感觉良好的场景。以下情况必须用独立 subagent review：

- 新增核心业务逻辑
- 修改了架构或关键接口
- 涉及安全、性能、可靠性的改动
- 代码量超过 200 行的大改动
- 你自己觉得"应该没问题"但又没十足把握的时候
- 用户明确要求 review 的代码

叫 subagent review 时：
- 把完整代码 + 需求背景 + 测试结果都传过去
- 不要说"我觉得没问题，你帮我过一下"（这是引导）
- 直接说"用 anti-overfit-reviewer 标准审查这段代码"

## 六、目标一致性检查（多轮对话必查）

审查代码前，先确认方向对不对：

- [ ] 回溯用户最初的目标是什么？
- [ ] 现在做的功能和最初目标一致吗？
- [ ] 有没有功能是后来加的，但和核心目标关系不大？
- [ ] 如果用户没说最近那些话，还会做这些功能吗？
- [ ] 最初的约束（时间、资源、范围）还在吗？

如果发现代码做了很多和最初目标无关的功能，即使代码质量再好，也要指出目标漂移问题。

## 七、阻塞级问题判定

出现以下任一情况，直接判 FAIL：

1. 修改测试代码来适配错误的实现
2. 测试只验证表面现象（文件存在/非空），完全不测核心逻辑
3. 多模块功能只有单元测试，完全没有集成验证，且集成风险高
4. 功能是 stub/placeholder，声称已完成但实际不可用
5. 错误处理全是 try-catch 吞掉，没有任何真实处理
6. 声称的性能/效果完全没有数据支持
7. 严重偏离用户最初目标，被最近几轮对话带偏了方向
