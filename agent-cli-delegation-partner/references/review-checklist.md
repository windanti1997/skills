# Codex 委托后 Review 清单

执行 Agent 完成后，Codex 必须先检查证据，再决定是否交付。

## 范围

- 结果是否满足用户真实需求？
- 是否遵守非目标？
- 是否引入了无关重构或大面积格式化？
- 是否把一个小任务扩大成了架构改造？

## 项目工作流

- 如果存在 Trellis，`.trellis/tasks/` 是否被正确更新？
- 如果存在 Superpowers / using-superpowers，是否遵守了项目自己的流程？
- 如果存在 spec / plan，代码和文档是否一致？
- 是否强行创建了项目不需要的流程文件？

## 代码与文档

- 改动文件是否符合预期？
- 文档是否是具体内容，而不是泛泛而谈？
- 边界条件是否放在正确层处理？
- 是否保留了用户已有改动？

## 验证

- 是否运行了有意义的测试、构建或静态检查？
- 失败是否如实报告？
- 如果跳过检查，理由是否成立？

## 安全与风险

- 没有未经授权的 commit、push、PR。
- 没有破坏性清理或回滚。
- 没有把 secrets 写进日志或文档。
- 没有未说明的迁移、权限、数据兼容风险。

## 决策

Codex 最后只能选择：

- 接受并向用户汇报。
- 再委托一个更窄的修复任务。
- 自己做小范围收尾。
- 停下来问用户，因为剩余问题属于产品/架构决策。
