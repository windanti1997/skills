# 第一性原理：为什么长任务会退化成 MVP

## 1. Agent 的默认目标通常不是用户真实目标

Agent 的短期优化目标通常是：给出一个看起来完整、风险较小、容易解释通过的交付物。

短任务中，这个代理目标和用户目标经常一致；长任务中，它们会分裂。

用户真实目标可能是：可交付、可维护、可验证、可扩展、能接真实调用链。

Agent 替代目标可能是：先跑通主流程、先做 MVP、先写壳子、后续补齐。

## 2. 长任务的成功标准模糊，模糊会被压缩

当成功标准没有硬证据时，Agent 会自然压缩范围：

- “完整”变成“基本可用”；
- “接入真实链路”变成“留接口”；
- “严格验收”变成“运行无报错”；
- “边界场景”变成“happy path”；
- “后续考虑”变成“本次逃避”。

## 3. 目标衰减是逐步发生的

长任务不是一次性失败，而是每一步都做局部合理降级：

```text
原始目标：完整能力 + 真实接入 + 严格验收
中途目标：先跑通核心流程
最终结果：壳子 + TODO + 文档解释 + 宽松验收
```

每一步都看似合理，但整体已经偏离目标。

## 4. 自审天然放水

同一个 Agent 生成任务、执行任务、验收任务时，会倾向于解释自己已经完成。

因此任务契约必须在实现前被审查；验收必须基于证据，而不是基于 Agent 声明。

## 5. Trellis 提供结构，不自动保证质量

Trellis 能让任务、实现记录、验证流程持久化，但如果任务契约很软，Agent 只会在 Trellis 里“结构化偷懒”。

核心不是“有没有 task”，而是 task 是否不可逃逸。

## 6. 真正的目标函数

一个好任务应该满足：

> 小到 Agent 能完成，硬到 Agent 不能糊弄，完整到用户能验收。
