# 测试契约：单测、集成测试与联调

> 测试不是可选的附属品，而是任务契约的核心组成部分。
> 没有测试方案的任务，就是不合格的任务。

## 1. 为什么需要测试契约

Agent 写代码最容易偷懒的地方就是测试：
- 只写"有测试"三个字，不写测了什么
- 测试只验证"函数能跑通"，不验证逻辑正确性
- 只有单元测试，没有集成测试，模块拼起来就炸
- 为了通过测试而写测试，不是为了验证功能
- 只测 happy path，不测边界和错误

测试契约的目标：让测试从"有没有"变成"够不够"，从"装饰"变成"保障"。

## 2. 测试的三个层次

### 2.1 单元测试：验证单个模块的逻辑正确性

**目标：** 确保单个函数/类/模块的逻辑是对的。

**必须覆盖：**
- 正常流程
- 边界条件（空值、极值、零值、最大最小值）
- 非法输入
- 错误路径（异常抛出、错误返回码）
- 状态转换（如果有状态机）

**禁止：**
- 只验证"函数返回非空"，不验证具体内容
- 只验证"不报错"，不验证结果正确性
- 为了凑覆盖率而写无意义的测试
- 测试和实现写得一模一样，等于自己证明自己

**断言深度要求：**
- ❌ 坏：`expect(result).not.toBeNull()`
- ❌ 坏：`expect(typeof result).toBe('object')`
- ✅ 好：`expect(result.status).toBe('error')`
- ✅ 好：`expect(result.message).toContain('required field')`
- ✅ 好：`expect(result.items).toHaveLength(3)`

### 2.2 集成测试：验证模块间的交互

**目标：** 确保多个模块拼起来能正确工作。

**什么时候必须有集成测试：**
- 任务涉及 2 个或以上模块的交互
- 有明确的调用链或数据流
- 模块间有接口契约
- 单个模块测试通过但整体可能失败

**必须覆盖：**
- 端到端完整流程（从输入到输出）
- 模块间接口契约
- 数据流向正确性
- 错误传播和处理
- 性能瓶颈点（如果有性能要求）

**禁止：**
- 只有单元测试，声称"集成没问题"
- 集成测试只测"不报错"，不测数据正确性
- 用 mock 把所有依赖都 mock 掉，等于又变回单元测试

### 2.3 联调集成测试：验证完整闭环

**目标：** 确保所有相关模块连起来能形成完整的用户可感知的闭环。

**什么时候需要单独建联调任务：**
- 任务拆成了多个子任务，每个子任务负责不同模块
- 模块之间有复杂的接口和数据流
- 单个模块测试通过不代表整体能工作
- 用户能感知的是完整功能，不是单个模块
- 依赖未实现的模块，需要等所有模块就绪后统一验证

**联调任务的特点：**
- 不写新功能，只做集成验证
- 发现问题后，分派回对应模块的子任务修复
- 验证的是完整用户流程，不是单个模块
- 必须有明确的通过/失败标准

## 3. 苏格拉底提问式测试设计

设计测试方案时，必须用苏格拉底提问拷问自己。答不上来的，测试方案不合格。

### 3.1 关于测试深度的提问

1. **这个测试真的验证了逻辑，还是只验证了"函数能跑通"？**
   - 如果我把核心逻辑改错了，这个测试会失败吗？
   - 如果我把返回值改成永远正确的固定值，测试还能通过吗？

2. **断言验证的是具体输出内容，还是只验证"不报错/返回非空"？**
   - 断言检查了几个具体属性？
   - 如果输出内容是错的但格式对，测试能发现吗？

3. **有没有为了通过测试而写测试的嫌疑？**
   - 测试是从需求推导出来的，还是从实现反推出来的？
   - 如果换一个人来写实现，这些测试还有意义吗？

### 3.2 关于测试覆盖的提问

4. **边界条件、失败路径、异常输入都覆盖了吗？**
   - 空值、零值、最大值、最小值都测了吗？
   - 错误抛出、错误返回、异常处理都测了吗？
   - 并发、时序、竞态条件（如果相关）都考虑了吗？

5. **只测单个函数有意义吗？模块拼起来会不会炸？**
   - 模块之间的接口契约验证了吗？
   - 数据在模块间流转正确吗？
   - 错误在模块间传播正确吗？

6. **这些测试通过了，能说明功能真的可用吗？**
   - 用户能感知到这些测试的价值吗？
   - 如果所有测试都通过了，但功能实际上不能用，可能是哪里漏了？

### 3.3 关于测试完整性的提问

7. **有没有为了凑覆盖率而写无意义的测试？**
   - 每个测试都有明确的测试目的吗？
   - 删掉这个测试，会影响对功能的信心吗？

8. **漏掉了什么最重要的测试场景？**
   - 最可能出问题的地方是什么？
   - 用户最可能踩的坑是什么？
   - 如果这个功能出 bug，最可能是哪里出？

## 4. 测试覆盖范围文档模板

每个任务的测试覆盖范围必须明确写入任务契约和最终完成报告。

### 任务契约中的测试范围（实现前写）

- [ ] 正常流程
- [ ] 边界条件（空值、极值、零值）
- [ ] 非法输入
- [ ] 错误路径（异常抛出、错误返回）

```markdown
## 测试覆盖范围

### 单元测试
- 已覆盖：
  - moduleA.functionX：正常流程、空值、非法输入、错误抛出
  - moduleB.functionY：正常流程、边界值、状态转换
- 未覆盖及原因：
  - 并发场景：当前版本不涉及并发，后续需要时补充

  - moduleA → moduleB → moduleC 完整调用链
  - 接口契约验证
  - 错误传播验证
- 未覆盖及原因：
  - 性能测试：当前没有性能要求，后续有需要再补充

### 测试运行结果
- 单测命令：[命令]
- 单测结果：[X passed, Y failed]
- 集成测试命令：[命令]
- 集成测试结果：[X passed, Y failed]
```

## 5. 联调集成测试任务设计模板

当任务涉及多模块协作时，单独建立联调任务。

```markdown
# Task: 联调集成测试 - [功能名称]

## 1. 背景与目标

本任务不开发新功能，专门用于验证以下模块联调后能形成完整闭环：

- 模块 A：[子任务 ID / 名称]
- 模块 B：[子任务 ID / 名称]
- 模块 C：[子任务 ID / 名称]

目标：确保 [完整用户流程] 能端到端跑通。

## 2. 前置依赖

- [ ] 模块 A 开发完成并通过测试
- [ ] 模块 B 开发完成并通过测试
- [ ] 模块 C 开发完成并通过测试

## 3. 联调验证范围

### 3.1 完整闭环验证

- [ ] 用户从 [起点] 到 [终点] 的完整流程
- [ ] 数据在各模块间正确流转
- [ ] 各模块接口契约一致

### 3.2 关键场景验证

- [ ] 正常流程
- [ ] 错误处理流程
- [ ] 边界条件
- [ ] 异常输入

## 4. 验收标准

- [ ] 完整用户流程能端到端跑通
- [ ] 各模块接口契约一致，没有不匹配
- [ ] 数据流转正确，没有丢失或错乱
- [ ] 错误能在正确的层级被处理
- [ ] 发现的问题都有对应修复任务

## 5. 问题跟踪

| 问题描述 | 所属模块 | 严重程度 | 状态 | 修复任务 |
|---|---|---|---|---|
| | | | | |
```

## 6. 常见测试偷懒模式与识别

### 模式 1：表面测试

**表现：** 测试只验证"函数存在""返回非空""不报错"，完全不测具体逻辑。

**识别：** 把核心逻辑改成错的，测试还能通过。

### 模式 2：Happy Path 独苗

**表现：** 只测正常流程，不测边界、错误、异常。

**识别：** 测试用例全是正常输入，没有任何错误场景。

### 模式 3：单元测试孤岛

**表现：** 每个模块的单测都通过了，但模块拼起来根本不工作。

**识别：** 多模块任务只有单元测试，没有集成测试。

### 模式 4：测试适配实现

**表现：** 实现写错了，不去修复实现，反而修改测试让它通过。

**识别：** 测试断言和错误的实现保持一致。

### 模式 5：凑覆盖率

**表现：** 为了达到覆盖率指标，写了很多无意义的测试。

**识别：** 测试只是调用函数，不验证任何有意义的输出。

### 模式 6：全 Mock 集成测试

**表现：** 说是集成测试，但把所有依赖都 mock 掉了，等于还是单元测试。

**识别：** 测试中 mock 了 3 个以上的依赖模块。

---

记住：测试的价值不在于"有多少个测试"，而在于"能发现多少问题"。
如果核心逻辑写错了测试都发现不了，那这些测试等于摆设。
