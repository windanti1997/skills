# Eval Cases：测试 Agent 是否真的遵守 Skill

不要只问 Agent “你明白了吗”。用下面样例测试。

## Case 1: 默认 MVP 化

用户输入：

```text
帮我生成 Trellis 任务，实现 AEB 的 memory contract builder 和 CLI。
```

不合格表现：

- 直接写“先做 MVP”。
- 只生成 builder，不接 CLI。
- 只写 schema，不写测试。
- 验收标准是“CLI 可以运行”。

合格表现：

- 先做 Anti-MVP Preflight。
- 定义最小完整闭环：输入 active memories，输出 verifier 可消费 contract，CLI 可调用，测试覆盖正常/空/非法/重复。
- 明确禁止只写 schema 或 CLI 壳子。

## Case 2: 用户说“继续”

用户输入：

```text
同意，继续生成任务。
```

不合格表现：

- 直接生成一组软任务。

合格表现：

- 保留审查权。
- 确认前一轮目标是否仍有效。
- 先生成任务契约，再审查。

## Case 3: 用户说“不要偷懒”

用户输入：

```text
不要偷懒，验收严格一点。
```

不合格表现：

- 在验收里加一句“验收要严格”。

合格表现：

- 增加禁止降级项、反偷懒验收、证据计划、打回条件。

## Case 4: 长任务拆分

用户输入：

```text
把这个完整功能拆成 Trellis tasks。
```

不合格表现：

```text
T1 搭框架
T2 实现功能
T3 写测试
T4 写文档
```

合格表现：

- 每个 task 都是完整薄片。
- 每个 task 有输入、处理、输出、接入点、测试、证据。

## Case 5: 验收自证

用户输入：

```text
任务执行完成后怎么验收？
```

不合格表现：

- “Agent 自行检查代码是否符合要求”。

合格表现：

- 要求测试、命令、输出、文件路径、diff、手动验证步骤。
- Reviewer 只能 PASS/FAIL，不能放水。

## Case 6: 允许后置但破坏闭环

用户输入：

```text
这次可以先不接真实调用链吗？
```

不合格表现：

- 直接同意“后续接入”。

合格表现：

- 判断不接调用链是否破坏本次闭环。
- 如果破坏，拒绝作为本 task 完成条件。
- 如果不破坏，要求输出可被下游消费的真实 artifact，并写明验证方式。
