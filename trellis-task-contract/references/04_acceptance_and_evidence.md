# 验收与证据规则

## 1. 验收项必须可判定

软验收：

```text
- 功能基本可用
- 页面正常展示
- 逻辑完善
- 测试通过
```

硬验收：

```text
- 当输入缺少 required 字段时，CLI 返回非 0 状态码，并输出包含字段名的错误信息。
- 新增 tests/test_contract_builder.py，覆盖正常输入、空输入、重复 memory id、非法 schema 四种场景。
- 运行 `pytest tests/test_contract_builder.py` 通过。
- CLI 输出的 contract 文件可被 verifier 读取，字段包含 obligations、source_memories、scope、created_at。
```

## 2. 每个验收项都要绑定证据

证据类型包括：

- 测试文件路径
- 运行命令
- 命令输出摘要
- 修改文件路径
- 日志片段
- 截图
- 生成产物路径
- 手动验证步骤

## 3. 反偷懒验收必须固定存在

每个任务都要有：

```markdown
## 反偷懒验收

- [ ] 没有未经授权降级成 MVP。
- [ ] 没有 TODO / placeholder / mock 冒充完成。
- [ ] 没有只做 happy path。
- [ ] 没有只写接口不接入真实调用链。
- [ ] 没有只写文档不实现。
- [ ] 每个验收项都有证据。
- [ ] 测试覆盖范围明确，不是只写"有测试"。
- [ ] 单测覆盖核心逻辑和边界，不是只测表面。
- [ ] 集成测试覆盖模块交互，不是只有单元测试孤岛。
```

## 4. 长任务证据回填

最终完成报告必须包含：

```markdown
## Evidence Report

### 修改文件

### 新增/更新测试

### 运行命令

### 命令结果

### 验收项到证据映射

| 验收项 | 证据 | 是否通过 |
|---|---|---|

### 未覆盖风险

### TODO / placeholder / mock 检查

### 测试覆盖范围

#### 单测覆盖
- 已覆盖：
- 未覆盖及原因：

#### 集成测试覆盖
- 已覆盖：
- 未覆盖及原因：
```

## 5. 不允许的完成声明

以下表达不能作为完成证据：

- "应该可以"
- "基本完成"
- "逻辑已完善"
- "后续可以补充"
- "已预留接口"
- "理论上支持"
- "未实际运行，但代码看起来没问题"
- "有测试"（必须说明测了什么、没测什么）
- "测试通过"（必须说明哪些测试、覆盖了什么场景）

## 6. 测试证据的深度要求

测试证据不能只给文件名和"测试通过"四个字。必须说明：

### 6.1 单元测试证据要求

- 测试了哪些具体函数/模块
- 覆盖了哪些场景（正常/边界/错误/非法输入）
- 断言验证了什么具体内容（不是只验证"不报错"）
- 核心逻辑写错了能不能被发现（反向验证）

### 6.2 集成测试证据要求

- 验证了哪些模块之间的交互
- 覆盖了哪条完整调用链
- 端到端流程的输入和输出分别是什么
- 接口契约是否符合预期

### 6.3 测试覆盖范围文档

任务完成报告中必须明确列出测试覆盖范围。只写"测试通过"而不说明覆盖范围的，视为测试证据不足。

## 7. 联调集成测试任务

涉及多模块协作的项目，必须有专门的联调集成测试环节。

### 什么时候需要联调任务

- 任务拆成了多个子任务，每个子任务负责不同模块
- 模块之间有复杂的接口和数据流
- 单个模块测试通过不代表整体能工作
- 用户能感知的是完整功能，不是单个模块

### 联调任务的目标

- 验证所有模块连起来能形成完整闭环
- 发现模块间接口不匹配的问题
- 验证端到端用户流程
- 确保数据在各模块间正确流转

### 联调任务的验收标准

- [ ] 完整用户流程能跑通
- [ ] 各模块接口契约一致
- [ ] 数据流转正确，没有丢失或错乱
- [ ] 错误能在正确的层级被处理
- [ ] 性能在可接受范围内（如果有性能要求）

## 8. 测试苏格拉底自检

设计测试方案和审查测试时，必须用苏格拉底提问拷问：

1. 这个测试真的验证了逻辑，还是只验证了"函数能跑通"？
2. 如果核心逻辑写错了，这个测试能发现吗？
3. 断言验证的是具体输出内容，还是只验证"不报错/返回非空"？
4. 边界条件、失败路径、异常输入都覆盖了吗？
5. 只测单个函数有意义吗？模块拼起来会不会炸？
6. 这些测试通过了，能说明功能真的可用吗？
7. 有没有为了凑覆盖率而写无意义的测试？
8. 漏掉了什么最重要的测试场景？

如果这些问题答不上来，说明测试方案还不够扎实。
