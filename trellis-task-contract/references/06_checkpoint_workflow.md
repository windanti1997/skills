# Checkpoint Workflow

长任务必须拆成 checkpoint，每个 checkpoint 都要有进入条件、完成条件和证据。

## 为什么需要 checkpoint

长任务如果只在最后验收，会出现：

- 前面已经偏了，最后很难纠正；
- Agent 用大量解释掩盖缺失；
- 测试和文档最后补，质量很差；
- 用户发现问题时返工成本很高。

## Checkpoint 模板

```markdown
## Checkpoint <N>: <名称>

### 目标

### 开始前条件

### 必须完成

### 禁止行为

### 证据

- 修改文件：
- 测试：
- 命令：
- 输出：

### 进入下一 checkpoint 的条件

### 如果不满足怎么办
```

## 推荐拆法

### 技术实现类任务

1. Contract / schema / 数据结构
2. 核心逻辑
3. 真实调用链接入
4. 测试与边界场景
5. 文档与最终证据报告

### 文档/文章类任务

1. 真实目标与读者入口
2. 结构与观点契约
3. 完整初稿
4. 反 AI 味 / 反讨好审查
5. 渠道适配与发布检查

### Agent 工作流类任务

1. 能力边界与控制面
2. 任务契约
3. 主执行 Agent prompt
4. Reviewer Agent prompt
5. Eval cases
6. 使用说明与安装路径

## 错误拆法

不要这样拆：

```text
T1: 搭框架
T2: 补功能
T3: 补测试
T4: 补文档
```

这会让 T1 变成壳子化假完成。

正确拆法是每个任务都是完整薄片：

```text
T1: 完成一个真实可运行的端到端闭环
T2: 扩展第二类场景并补齐测试
T3: 接入真实调用链并做回归验证
T4: 清理旧逻辑并更新文档
```
