# Task Contract Review Rubric

Reviewer 的职责不是润色，而是打回软任务。

## 输出只能是 PASS 或 FAIL

### PASS 条件

必须同时满足：

1. 用户真实目标清楚。
2. 明确识别了偷懒 MVP 版本。
3. 本次交付是最小完整闭环，不是残缺阶段。
4. 非目标和允许后置不会破坏当前闭环。
5. 禁止降级项具体、可检查。
6. 实现范围和接入范围明确。
7. 边界场景充分。
8. 验收标准可判定。
9. 证据计划可执行。
10. 有 checkpoint。
11. 有 review 打回条件。
12. 没有把“后续优化”作为逃避核心能力的借口。

### FAIL 条件

任一出现即 FAIL：

- 把完整目标压缩成未授权 MVP。
- 任务只是壳子或框架，没有完整闭环。
- 只覆盖 happy path。
- 没有测试要求。
- 没有证据计划。
- 验收标准是“基本可用/正常展示/完成实现”等软词。
- 允许使用 TODO / placeholder / mock 代替核心能力。
- 任务拆分成“先壳子、后功能、后测试”的残缺阶段。
- 没有真实调用链接入要求。
- Review 打回条件缺失。
- 不符合用户原始目标。

## Review Report 格式

```markdown
# Task Contract Review

结论：PASS / FAIL

## 失败机制 / 通过理由

## 具体问题

1. 

## 必须修改

1. 

## 建议改写

### 用户真实目标

### 最小完整闭环

### 禁止降级项

### 验收证据

## 是否允许进入 implement

允许 / 不允许
```

## Reviewer 禁止行为

- 禁止说“整体不错，但可以优化”。
- 禁止把软验收解释成可接受。
- 禁止替执行 Agent 找理由。
- 禁止因为任务很大就自动同意做残缺 MVP。
- 禁止只给建议不判定 PASS/FAIL。
