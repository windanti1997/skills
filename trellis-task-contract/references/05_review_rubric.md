# Task Contract Review Rubric

Reviewer 的职责不是润色，而是打回软任务。

## 输出只能是 PASS 或 FAIL

### PASS 条件

必须同时满足：

1. 用户真实目标清楚。
2. 明确识别了偷懒 MVP 版本。
3. 本次交付是最小完整闭环，不是残缺阶段。
4. 非目标和允许后置不会破坏当前闭环。
5. 禁止降级项具体、可检查。
6. 实现范围和接入范围明确。
7. 边界场景充分。
8. 测试覆盖范围明确（单测 + 集成测试，测什么、不测什么都写清楚了。
9. 单测覆盖核心逻辑和边界，不是只测表面。
10. 集成测试覆盖模块交互，不是只有单元测试孤岛。
11. 多模块任务有联调集成测试方案。
12. 验收标准可判定。
13. 证据计划可执行。
14. 有 checkpoint。
15. 有 review 打回条件。
16. 没有把"后续优化"作为逃避核心能力的借口。
17. 测试方案通过了苏格拉底自检（核心逻辑写错了能被发现吗？）。

### FAIL 条件

任一出现即 FAIL：

- 把完整目标压缩成未授权 MVP。
- 任务只是壳子或框架，没有完整闭环。
- 只覆盖 happy path。
- 没有测试要求，或只写了"有测试"三个字。
- 测试只测表面现象（文件存在/函数返回非空），不测核心逻辑。
- 单测只有 happy path，没有边界条件和错误路径。
- 多模块任务只有单元测试，没有集成测试方案。
- 多模块任务没有联调集成测试任务。
- 没有证据计划。
- 验收标准是"基本可用/正常展示/完成实现"等软词。
- 允许使用 TODO / placeholder / mock 代替核心能力。
- 任务拆分成"先壳子、后功能、后测试"的残缺阶段。
- 没有真实调用链接入要求。
- Review 打回条件缺失。
- 不符合用户原始目标。
- 测试方案答不上苏格拉底自检问题。

## 测试专项审查维度

### 1. 单测深度审查

- [ ] 单测覆盖了哪些具体模块/函数？
- [ ] 断言验证的是具体输出内容，还是只验证"不报错/返回非空"？
- [ ] 如果核心逻辑写错了，测试能发现吗？
- [ ] 边界条件、错误路径、非法输入都覆盖了吗？
- [ ] 有没有为了凑覆盖率而写无意义的测试？

### 2. 集成测试审查

- [ ] 涉及多模块交互的功能，有没有集成测试？
- [ ] 集成测试验证了哪条完整调用链？
- [ ] 端到端流程有没有人能跑通？
- [ ] 接口契约有没有验证？
- [ ] 数据在模块间流转正确吗？

### 3. 测试苏格拉底审查

用苏格拉底提问拷问测试方案：

1. 这个测试真的验证了逻辑，还是只验证了"函数能跑通"？
2. 如果核心逻辑写错了，这个测试能发现吗？
3. 断言验证的是具体输出内容，还是只验证"不报错/返回非空"？
4. 边界条件、失败路径、异常输入都覆盖了吗？
5. 只测单个函数有意义吗？模块拼起来会不会炸？
6. 这些测试通过了，能说明功能真的可用吗？
7. 有没有为了凑覆盖率而写无意义的测试？
8. 漏掉了什么最重要的测试场景？

答不上来的，测试方案不扎实，打回重写。

## Review Report 格式

```markdown
# Task Contract Review

结论：PASS / FAIL

## 失败机制 / 通过理由

## 具体问题

1. 

## 必须修改

1. 

## 建议改写

### 用户真实目标

### 最小完整闭环

### 禁止降级项

### 测试覆盖范围

### 验收证据

## 是否允许进入 implement

允许 / 不允许
```

## Reviewer 禁止行为

- 禁止说"整体不错，但可以优化"。
- 禁止把软验收解释成可接受。
- 禁止替执行 Agent 找理由。
- 禁止因为任务很大就自动同意做残缺 MVP。
- 禁止只给建议不判定 PASS/FAIL。
- 禁止因为"有测试文件"就放过，必须检查测试深度。
- 禁止因为"测试通过了"就放过，必须检查测试覆盖了什么。
