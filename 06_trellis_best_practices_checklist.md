## 🏃 快速上手 5 步走

### 第 1 步：安装 & 初始化（5 分钟）

```bash
# 安装 Trellis CLI
npm install -g @mindfold/trellis-cli

# 在项目根目录初始化
cd your-project
trellis init
```

✅ **检查清单**：
- [ ] 安装成功，`trellis --version` 能输出版本号
- [ ] `.trellis/` 目录已创建
- [ ] 配置了默认使用的 AI 平台（如 claude-code）

---

### 第 2 步：从现有代码提取第一版 Spec（30 分钟）

**不要自己从零写 spec！** 让 AI 从真实代码里提取。

```bash
# 让 AI 扫描代码库，生成第一版规范
# （在你的 AI 编程助手里执行）
"请扫描这个项目的代码结构，从真实代码中提取
 后端编码规范，生成 trellis spec。
 每条规范必须能引用具体的源文件作为例子。"
```

✅ **检查清单**：
- [ ] 生成了 3-5 个核心 spec 文件（不是 20 个）
- [ ] 每条规范都有对应的源码例子
- [ ] 没有"想当然"的理想规范

💡 **小技巧**：先只做最核心的 2-3 个模块，比如后端 API 规范、前端组件规范。贪多嚼不烂。

---

### 第 3 步：找一个小需求跑通完整流程（2 小时）

选一个**小而完整**的需求，比如：
- 加一个新的 API 接口
- 加一个新页面
- 改一个独立的小功能

完整走一遍 Trellis 流程：

```
1. trellis task:new → 创建任务
2. 写 prd.md → 明确目标、范围、验收标准
3. implement → 按规范实现
4. check → 对照 spec 检查
5. finish → 完成任务，更新 workspace 记忆
```

✅ **检查清单**：
- [ ] 任务有明确的 in-scope / out-of-scope
- [ ] 实现过程中 AI 主动参考了 spec
- [ ] check 阶段发现了 1-2 个不符合规范的地方
- [ ] 任务结束后 journal 自动更新了

---

### 第 4 步：补充和修正 Spec（持续进行）

做完第一个任务后，你会发现：
- 有些 spec 写得不对
- 有些场景 spec 没覆盖到
- 有些规范太模糊，AI 理解不了

这很正常！**spec 是长出来的，不是设计出来的。**

```bash
# 发现新规范时，及时补充
"这个模式很好，把它加到 spec 里。"

# 发现 spec 不对时，及时修正
"这条规范跟实际代码不符，更新一下。"
```

✅ **检查清单**：
- [ ] 每周至少更新 1-2 次 spec
- [ ] 每条 spec 更新都有对应的真实场景
- [ ] 删除了那些"从来没用到过"的 spec

---

### 第 5 步：享受复利效应（第 2 周开始）

到第 2 周，你会明显感觉到：
- 开新会话不用重新讲背景了
- AI 写的代码风格越来越统一
- Review 时"不符合规范"的评论少了很多

这就是 Trellis 的复利：**用得越多，AI 越懂你的项目。**

## 📐 Spec 编写 7 条铁律

### ❌ 反面教材（别这么写）

```markdown
## 代码规范
- 代码要整洁、规范
- 命名要有意义
- 注释要清晰
```

💩 全是正确的废话，AI 看了也不知道怎么做。

---

### ✅ 正确写法（7 条铁律）

#### 铁律 1：具体，不要抽象

**坏**："错误处理要规范"
**好**："所有 API 错误必须返回 `{ code: string, message: string }` 格式，code 用大写下划线命名，如 `USER_NOT_FOUND`"

#### 铁律 2：有正反例

```markdown
// ❌ Bad
const user = await userService.create(req.body);

// ✅ Good
const input = CreateUserSchema.parse(req.body);
const user = await userService.create(input);
```

#### 铁律 3：说清"为什么"

```markdown
## 为什么必须用 Zod 校验？
- 防止未经验证的数据进入 service 层
- 自动生成类型定义，前后端一致
- 错误信息标准化，前端可以直接展示
```

AI 理解了为什么，遇到新场景才能正确判断。

#### 铁律 4：能追溯到源文件

```markdown
参考实现：
- src/modules/user/user.route.ts 第 20-35 行
- src/modules/auth/auth.route.ts 第 15-28 行
```

有具体的代码锚点，AI 可以自己去看完整上下文。

#### 铁律 5：按模块拆分，不要写大文件

```
.trellis/spec/
├── backend/
│   ├── api-validation.md     ← 单一主题
│   ├── error-handling.md     ← 单一主题
│   └── database-guidelines.md
├── frontend/
│   ├── component-guidelines.md
│   └── state-management.md
└── guides/
    └── testing-practices.md
```

每个 spec 文件只讲一件事，Trellis 会按需加载，不浪费上下文。

#### 铁律 6：用"我们"的语气，不是"你应该"

**坏**："你应该使用 Zod 进行参数校验"
**好**："我们的项目使用 Zod 进行参数校验，所有 route 层必须先校验再调用 service"

这是团队的共同约定，不是自上而下的命令。

#### 铁律 7：能自动化检查的，就不要写在 spec 里

如果 ESLint、Prettier、TypeScript 已经能检查了，就不用再写进 spec。
Spec 只放那些**工具检查不了、需要人判断**的规范。

## 💣 10 个最容易踩的坑

### 坑 1：一开始就想写完整的 spec
**症状**：初始化第一天就填了 20 个模板文件
**后果**：一半是脑补的，跟实际代码对不上
**正确姿势**：从真实代码提取，用到哪写到哪

---

### 坑 2：什么任务都走完整 Trellis 流程
**症状**：改个文案也要建 task、写 prd、走 check
**后果**：小任务 overhead 比收益还大
**正确姿势**：
- 简单对话 → 不用 Trellis
- 小改动 → inline 模式
- 多文件、跨模块、需要持久规划 → 才建完整 task

---

### 坑 3：spec 写得太抽象
**症状**："代码要整洁""命名要有意义"
**后果**：AI 看了等于没看，该怎么错还怎么错
**正确姿势**：每条规范必须有正反例、有具体路径

---

### 坑 4：不更新 workspace journal
**症状**：任务结束就结束了，不记录做了什么
**后果**：下次开新会话还是要从头讲
**正确姿势**：每次任务结束确认一下 journal，确保关键信息都记下来了

---

### 坑 5：把 spec 当圣经，不允许修改
**症状**：spec 写了就不能改，改了就是"不规范"
**后果**：spec 越来越僵化，跟实际代码脱节
**正确姿势**：spec 是活的，随时可以更新，更新本身就是最佳实践

---

### 坑 6：团队推广时强制所有人用
**症状**：一上来就要求全团队必须用 Trellis
**后果**：抵触情绪大，推广失败
**正确姿势**：先让 1-2 个愿意尝鲜的人用，跑出效果了再慢慢扩散

---

### 坑 7：不 Review spec
**症状**：spec 随便写，没人 Review
**后果**：质量参差不齐，反而误导 AI
**正确姿势**：像 Review 代码一样 Review spec，每条规范都要经过团队确认

---

### 坑 8：贪多求全，什么规范都想加
**症状**：spec 里塞了几十条，面面俱到
**后果**：AI 抓不住重点，重要的规范被淹没
**正确姿势**：优先加那些"反复踩坑"的规范，80% 的问题来自 20% 的场景

---

### 坑 9：只写"不能做什么"，不写"应该怎么做"
**症状**：spec 全是禁令，没有正面引导
**后果**：AI 知道不能做什么，但不知道该怎么做
**正确姿势**：每条"不要"的后面，都跟着一个"应该"的例子

---

### 坑 10：期望 Trellis 解决所有问题
**症状**：上了 Trellis 就想一劳永逸
**后果**：发现还是有问题，就觉得 Trellis 没用
**正确姿势**：Trellis 是辅助轮，不是自动驾驶。它减少重复劳动，但不替代人的判断

## ⌨️ 常用命令速查表

### 初始化 & 配置
| 命令 | 作用 |
|---|---|
| `trellis init` | 在当前项目初始化 Trellis |
| `trellis --version` | 查看版本号 |
| `trellis doctor` | 检查环境和配置是否正常 |

### 任务管理
| 命令 | 作用 |
|---|---|
| `trellis task:new` | 创建新任务 |
| `trellis task:list` | 列出所有任务 |
| `trellis task:switch <id>` | 切换到指定任务 |
| `trellis task:archive <id>` | 归档已完成的任务 |

### Spec 管理
| 命令 | 作用 |
|---|---|
| `trellis spec:list` | 列出所有 spec |
| `trellis spec:new <name>` | 创建新的 spec 文件 |
| `trellis spec:update` | 从当前任务更新 spec |

### 工作流
| 命令/操作 | 作用 |
|---|---|
| `/trellis:plan` | 进入规划阶段 |
| `/trellis:implement` | 进入实现阶段 |
| `/trellis:check` | 进入检查阶段 |
| `/trellis:finish` | 完成任务，更新记忆 |
| `/trellis:update-spec` | 将本次发现的新模式更新到 spec |

💡 **提示**：不同平台的触发方式可能不同，有的是 `/trellis:xxx` 命令，有的是自动 hook。具体看你用的平台。

## 👥 团队推广 4 步法

### 第 1 步：种子用户试点（第 1 周）
- 找 1-2 个愿意尝鲜的开发者
- 选一个中等复杂度的项目
- 目标：跑通流程，验证价值

### 第 2 步：沉淀核心 Spec（第 2-3 周）
- 从试点项目提取 5-10 条最有用的 spec
- 团队一起 Review，达成共识
- 目标：有一套大家认可的核心规范

### 第 3 步：小范围推广（第 4-6 周）
- 扩展到 3-5 个人的小团队
- 建立每周 spec 更新机制
- 收集反馈，持续优化
- 目标：证明团队协作场景的价值

### 第 4 步：全面推广（第 7 周起）
- 全团队接入
- 建立 spec Review 流程
- 新人 onboarding 加入 Trellis 培训
- 目标：成为团队标准工作流

## 🎯 不同场景的使用模式

### 场景 1：从零开始新项目
**模式**：先跑起来，边做边补 spec
- 第 1 周：先写核心技术栈的 spec（2-3 个文件）
- 第 2 周起：每做一个新模块，补充对应 spec
- **关键**：不要等 spec 写全了再开始写代码

### 场景 2：接入存量老项目
**模式**：从真实代码提取，不要凭空设计
- 第一步：让 AI 扫描代码库，生成第一版 spec
- 第二步：团队 Review，删掉脑补的，保留真实的
- 第三步：找一个小需求试点，边做边补
- **关键**：每条 spec 必须能找到源码依据

### 场景 3：修复反复出现的 Bug
**模式**：Bug → 根因分析 → 沉淀 spec → 不再犯
1. 修 Bug + 加回归测试
2. 做根因分析：为什么会发生？
3. 沉淀成 spec：以后怎么避免？
4. 加进 check 清单：自动检查
- **关键**：同样的坑，不能踩第二次

### 场景 4：新人入职 onboarding
**模式**：先读 spec，再做任务
1. 先读核心 spec 文件（半天）
2. 跟着 Trellis 流程做 2-3 个小任务
3. 做任务的过程中理解规范
- **关键**：从"人传人"变成"文档传"

## 💡 最后一句话

Trellis 不是银弹，它不会让你的 AI 突然变得超级厉害。

它做的事情很朴素：
- 把你的规矩记下来
- 把做过的事记下来
- 把一个人的经验，变成全队的资产

这件事不性感，但很重要。

**用得越久，价值越大。**
